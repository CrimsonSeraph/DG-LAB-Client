### （二）新功能

#### 1. 自定义波形
- **需求**：用户可设计自定义波形（非预设脉冲串）。
- **三种编辑方式对比**：

| 方式 | 优点 | 缺点 | 考虑程度 |
| - | - | - | - |
| 鼠标绘制（点阵） | 直观，适合非技术用户 | 难以精确控制，数据量大 | ⭐⭐⭐（作为辅助） |
| 函数表达式（如 `sin(t)*100`） | 精确，可参数化 | 需要学习表达式语法 | ⭐⭐⭐⭐（核心） |
| 关键帧点填数值 | 精确，适合离散脉冲 | 编辑繁琐 | ⭐⭐（可选） |

- **可能方案**：**函数表达式为主 + 鼠标绘制为辅**。表达式可生成连续波形，再采样为脉冲字符串；鼠标绘制用于微调。存储格式统一为脉冲字符串（兼容现有协议）。

#### 2. 数据模块支持（核心扩展）
- **目标**：从外部程序（游戏、传感器等）获取数据，经规则处理发送指令。
- **设计方案**（已整合疑问解答）：

| 设计点 | 方案 |
| - | - |
| 模块实现形式 | **Python 子进程 + TCP 通信**，与现有 Bridge 模式类似。每个数据模块独立运行，监听本地端口。 |
| 端口分配 | **每个数据模块使用独立端口**（由模块自身分配，主程序动态发现）。 |
| 模块标识与区分 | 每条消息包含唯一 `module_id`（如 `"csgo_health"`），主程序维护活跃模块列表。 |
| 数据顺序保证 | 模块发送**数组**，例如 `{"data": [100, 80, 30]}`。用户可按索引自定义显示名称。 |
| 数据不完整/丢失处理 | 模块发送错误状态 `{"status": "error", "message": "..."}`，主程序通知用户并暂停规则执行。 |
| 模块发现机制 | 主程序扫描指定目录（如 `modules/`）下的 Python 脚本，启动每个脚本并读取其分配端口；或采用配置文件静态注册。 |
| 性能与限制 | 每个模块独立进程 + TCP 开销稍大，建议限制最大并发模块数（如 10 个），并提供解除限制选项。 |

- **通信协议示例**：
  ```json
  // 模块 -> 主程序
  {
    "module_id": "csgo_health",
    "data": [100, 80, 30],
    "timestamp": 1712345678
  }
  // 主程序 -> 模块（可选，用于控制模块行为）
  {
    "cmd": "pause",
    "module_id": "csgo_health"
  }
  ```
- **用户配置**：在客户端“配置”页面中，可为每个数据索引自定义显示名称（如索引0 → “生命值”），名称存储在 `user.json` 中。

#### 3. 规则间结果引用
- **需求**：规则 A 的计算结果作为规则 B 的输入参数。
- **实现方式**：引入“临时变量”机制。每条规则执行后可选择将结果存入命名变量（如 `$temp`），后续规则可使用 `{$temp}` 占位符。注意避免循环引用。
- **复杂度**：较高，建议在数据模块完善后再考虑。

#### 4. 日志持久化
- **需求**：保留近期日志到文件，区分常规日志和错误日志。
- **方案**：
  - 常规日志：保留最近 3~5 次运行的日志文件（按时间命名），每次启动创建新文件。
  - 错误日志：独立文件，保留最近 10~20 次错误记录。
  - 使用 `DebugLog` 的 `FileSink`，可配置保留份数。

### （三）旧功能改进

#### 1. 日志级别优化
- **问题**：部分 DEBUG 信息级别过高，部分关键信息无日志。
- **方案**：重新审查所有 `LOG_MODULE` 调用，将调试信息降为 `LOG_DEBUG`，错误/警告保持原级别。补充重要操作日志（如规则加载成功、命令发送失败等）。

#### 2. 合并连接/断开按钮
- **需求**：将“启动连接”和“终止连接”合并为一个切换按钮（如“连接/断开”）。
- **实现**：使用 `QPushButton` 的 `setCheckable(true)`，根据状态改变文本和样式。

#### 3. 规则与配置的导入/导出
- **规则**：添加“导出规则文件”按钮，将当前规则保存为 JSON；添加“导入规则文件”，合并或替换现有规则。
- **配置**：已有 `MultiConfigManager` 支持，可在界面增加“导出配置”功能，将三个配置文件打包为 ZIP。

### （四）长期计划

#### 1. 移除 Python WebSocket 依赖
- **动机**：减少外部依赖，统一技术栈。
- **方案**：使用 C++ WebSocket 库（如 `QtWebSockets`、`uWebSockets`、`ixwebsocket`）重新实现 `WebSocketCore.py` 功能。
- **保留 Python**：仍保留 Python 子进程机制，仅用于支持未来可能的数据模块（Python 插件）。

#### 2. 用 C++ 实现 DG-Lab 官方 WebSocket 服务端
- **可行性**：官方服务端是开源的（Node.js），核心逻辑为连接管理、消息转发、心跳、波形定时器。用 C++ 实现难度中等，可借助 `boost::asio` 或 `libuv`。
- **价值**：提升性能，集成到客户端中作为内置服务（无需单独启动 Node.js）。
- **建议**：作为独立项目先实验，成熟后再合并。

## 三、个性化定制（UI 与主题）

### （一）Qt 界面个性化

#### 1. 主题自定义
- **整体颜色**：可通过 `QApplication::setPalette()` 动态修改，支持明/暗主题切换（已有 `style_light.qcss` / `style_night.qcss`）。不同页面不同颜色需为每个页面单独设置样式表。
- **按键属性**：在 `.qss` 中定义 `QPushButton { background-color: #xxx; border-radius: 5px; }`。
- **文字样式**：加载字体包使用 `QFontDatabase::addApplicationFont()`，然后设置 `QFont`。对一段文字的部分字符单独着色，可使用 `QTextDocument` 或 `QLabel` 配合 HTML。

#### 2. 杂项
- **自定义首页图片/背景**：使用 `QLabel` 设置 `QPixmap`，支持 GIF（`QMovie`），不支持 MP4（需用 `QMediaPlayer` + `QVideoWidget`，较重量）。
- **动画**：使用 `QPropertyAnimation` 或 `QGraphicsEffect`。
- **应用图标**：设计需体现“电子脉冲”+“连接”元素。可使用开源图标库（如 Font Awesome）或委托设计。
- **表格/控制台背景个性化**：表格背景未生效可能是因为样式被视图代理覆盖。可尝试在 `QTableView` 的 `setStyleSheet` 中使用 `::item` 伪状态，或重写 `paint` 方法。控制台（`QTextEdit`）直接设置 `background-color` 应有效。

#### 3. UI 设计优化
- **建议**：参考现代软件布局（左侧导航栏，右侧内容区）。可使用 `QStackedWidget` + `QListWidget` 模拟。学习 Qt 官方示例或 Figma 原型。

### （二）调试控制台个性化
- **更多颜色主题**：预置几套配色（如 One Dark、Solarized），用户通过下拉选择。
- **自定义颜色**：提供 RGB 选择器，动态生成样式表应用。
